# PRD: 닌자거북의홈 (HOMEninja)

## 목표
인디 앱 개발자의 모든 앱을 한 페이지에서 소개하고 다운로드할 수 있는 포트폴리오 사이트.
커뮤니티/게시판에 이 하나의 링크만 공유하면, 방문자가 앱을 발견하고 즉시 다운로드한다.

## 사용자
- **주 사용자**: 커뮤니티/게시판(클리앙, 뽐뿌, Reddit 등)에서 링크를 클릭해 들어오는 잠재 사용자
- **유입 경로**: 게시판 글 → 링크 클릭 → 홈페이지 도착
- **디바이스**: 모바일 70% 이상으로 예상 (게시판 앱/모바일 브라우저에서 유입)
- **체류 시간**: 짧음 (30초~1분). 빠르게 훑고 다운로드하거나 이탈
- **기술 수준**: 일반 사용자. 개발자가 아님

## 사용자 여정

### 여정 A: 홈페이지 전체 링크로 유입 (메인 시나리오)
```
1. 게시판에서 홈페이지 링크 클릭 (예: mysite.vercel.app)
   └─ 모바일 브라우저 또는 인앱 브라우저에서 열림
2. 히어로 섹션 도착
   └─ 누구의 사이트인지 인지 (0.5초)
   └─ 앱 바로가기 버튼 확인 → 관심 앱 클릭하여 바로 이동
   └─ 또는 스크롤 유도 확인 → 아래로 스크롤
3. 첫 번째 앱(행성) 도달
   └─ 앱 이름 + 한 줄 설명 읽기 (2초)
   └─ 스크린샷으로 UI 확인 (3초)
   └─ 관심 있으면 → 다운로드 버튼 탭
   └─ 관심 없으면 → 계속 스크롤
4. 두 번째 앱(행성) 도달
   └─ 동일 흐름
5. 푸터 도달
   └─ 연락처 확인 또는 이탈
```

### 여정 B: 특정 앱 앵커 링크로 유입 (핵심 시나리오)
```
1. 게시판에서 특정 앱 링크 클릭 (예: mysite.vercel.app/#memory-palace)
   └─ 해당 앱 섹션으로 바로 스크롤 (히어로 건너뜀)
2. 타겟 앱 섹션 도착
   └─ 앱 이름 + 설명 확인
   └─ 플랫폼 감지 → 내 디바이스에 맞는 다운로드 버튼이 강조됨
   └─ 다운로드 버튼 탭 → 스토어/웹앱으로 이동
3. 위/아래 스크롤하여 다른 앱 발견
   └─ 추가 다운로드 또는 이탈
```

### 여정 C: 소셜/메신저 공유 카드로 유입
```
1. 카카오톡/슬랙 등에서 OG 프리뷰 카드 확인
   └─ 제목 + 설명 + 프리뷰 이미지로 사이트 인지
2. 카드 탭 → 홈페이지 도착
   └─ 여정 A 또는 B와 동일
```

## 핵심 기능 상세

### F-001: 히어로 섹션
- 개발자 이름 또는 브랜드명 표시
- 한 줄 소개 (예: "앱으로 세상을 조금 편하게")
- **앱 바로가기 버튼**: 각 앱 이름을 나열하여 클릭 시 해당 섹션으로 smooth scroll
  - 앱이 2~3개이므로 네비게이션 바 대신 히어로 내 인라인 버튼으로 충분
  - 예: [기억의궁전] [제주택배비지원]
- 스크롤 유도 인디케이터 (아래 화살표 또는 텍스트)
- 배경: 별이 반짝이는 우주 (Canvas)

### F-002: 앱 소개 섹션 (행성 단위)
- 각 앱은 하나의 행성으로 표현
- 행성 그래픽이 섹션 배경에 대형으로 배치
- **앵커 ID**: 각 섹션에 `id={app.id}` 부여 → URL 해시로 직접 접근 가능 (예: `#memory-palace`)
  - 게시판에 `mysite.vercel.app/#memory-palace` 형태로 특정 앱 링크를 공유할 수 있음
  - 해시 링크 접근 시 해당 섹션으로 자동 스크롤
- 앱 이름 (큰 텍스트)
- 한 줄 설명
- **앱 부가 정보** (작은 텍스트, 선택적 표시):
  - 지원 OS 버전 (예: "iOS 16.0+", "Android 8.0+")
  - 최종 업데이트 날짜 (예: "2026.03 업데이트")
  - 이 정보들은 AppData에 optional 필드로 포함
- 스크린샷 1~2장 (디바이스 목업 없이 원본 그대로)
- 다운로드 버튼 (플랫폼별)

### F-003: 다운로드 버튼
- **iOS 앱**: App Store 배지 → 탭 시 App Store 페이지로 이동
- **Android 앱**: Google Play 배지 → 탭 시 Play Store 페이지로 이동
- **Web 앱**: 텍스트 버튼 ("바로 사용하기 →") → 탭 시 웹앱 URL로 이동
- 모든 외부 링크는 새 탭(`target="_blank"`, `rel="noopener noreferrer"`)으로 열기
- 링크가 아직 없는 앱: "준비 중" 상태 표시 (버튼 비활성화 + 텍스트)
- **플랫폼 감지**: 사용자 디바이스를 감지하여 관련 버튼을 시각적으로 강조
  - iPhone/iPad → App Store 버튼을 primary 스타일로, Google Play는 secondary
  - Android → Google Play 버튼을 primary 스타일로, App Store는 secondary
  - 데스크탑/기타 → 모든 버튼 동등하게 표시
  - 감지 실패 시 → 모든 버튼 동등하게 (fallback)
  - 감지 방법: `navigator.userAgent` (SSR에서는 불가하므로 클라이언트에서만)
  - 숨기는 게 아니라 강조 순서만 변경 — 모든 버튼은 항상 보임
- **App Store / Google Play 배지 가이드라인 준수**:
  - Apple: 공식 "Download on the App Store" SVG 배지 사용, 최소 크기 준수
  - Google: 공식 "Get it on Google Play" SVG 배지 사용, 최소 크기 준수
  - 두 배지의 높이를 동일하게 맞춤
  - 자체 디자인 버튼 대신 공식 배지 사용 (스토어 정책)

### F-004: 별 배경
- Canvas로 렌더링하는 별 파티클
- 별 개수: 뷰포트 크기에 비례 (모바일 ~150개, 데스크탑 ~300개)
- 반짝임 애니메이션 (opacity 변화)
- 스크롤 시 패럴랙스 (별이 콘텐츠보다 느리게 이동)

### F-005: 행성 그래픽
- CSS gradient + box-shadow로 렌더링
- 각 행성은 고유한 색상/크기/위치
- 스크롤 시 패럴랙스 (콘텐츠보다 더 느리게 이동)
- 행성은 순수 장식 — 클릭/인터랙션 없음

### F-006: 푸터
- 이메일 또는 연락 가능한 링크
- GitHub / SNS 링크 (선택)
- **개인정보처리방침 링크**: 앱스토어 등록 앱은 개인정보처리방침 URL 필수. 별도 페이지(`/privacy`) 또는 외부 링크
- 저작권 표시

### F-007: 앵커 링크 네비게이션
- 각 앱 섹션에 URL-safe한 id 부여 (예: `memory-palace`, `jeju-delivery`)
- `mysite.vercel.app/#memory-palace` 형태로 특정 앱에 직접 접근 가능
- 히어로의 앱 바로가기 버튼 클릭 시 해당 섹션으로 smooth scroll
- 브라우저 뒤로가기 시 히스토리 정상 동작

### F-008: 플랫폼 감지 (다운로드 UX 최적화)
- 사용자 디바이스에 맞는 다운로드 버튼을 시각적으로 강조
- iOS → App Store 우선, Android → Google Play 우선
- 모든 버튼은 항상 표시 (숨기지 않음)
- Client Component에서 `navigator.userAgent`로 감지
- SSR 시에는 감지 불가 → 기본값은 모든 버튼 동등 표시 → 하이드레이션 후 강조 적용
- 하이드레이션 mismatch 방지: 초기 렌더 시 모든 버튼 동등 → useEffect에서 감지 후 클래스 변경

### F-009: 웹 기본 인프라
- **Favicon**: 우주 테마에 맞는 아이콘 (app/favicon.ico 또는 app/icon.tsx로 동적 생성)
- **Apple Touch Icon**: iOS 홈 화면 추가 시 표시될 아이콘 (180×180)
- **Web Manifest**: PWA 아님. 하지만 기본 manifest로 홈 화면 추가 시 이름/아이콘 제공
- **sitemap.xml**: `app/sitemap.ts`로 자동 생성. 페이지 1개 (루트)만 포함
- **robots.txt**: `app/robots.ts`로 자동 생성. 모든 크롤러 허용

### F-010: 개인정보처리방침 페이지
- 경로: `/privacy`
- 앱스토어에 등록한 앱의 개인정보처리방침 URL로 사용
- 내용: 수집하는 개인정보, 이용 목적, 보관 기간, 연락처
- 우주 테마 적용 (별 배경은 layout.tsx에서 자동 적용)
- 디자인은 최소한 — 텍스트 중심

## 비기능 요구사항

### 성능
- Lighthouse Performance 점수 90 이상 목표
- First Contentful Paint (FCP) < 1.5초
- Largest Contentful Paint (LCP) < 2.5초
- Canvas 애니메이션 60fps 유지 (모바일 포함)
- 전체 페이지 크기 < 500KB (스크린샷 제외)
- 스크린샷은 WebP 포맷, 각 이미지 < 200KB

### SEO / 소셜 공유
- 페이지 title: "{이름} — 앱 포트폴리오"
- meta description: 앱 목록 요약
- Open Graph 태그 (og:title, og:description, og:image, og:url)
- Twitter Card 태그 (twitter:card=summary_large_image)
- OG 이미지: 우주 테마에 맞는 1200x630 프리뷰 이미지 (추후 제작)
- canonical URL 설정
- **sitemap.xml**: 루트 페이지 + /privacy 페이지 포함
- **robots.txt**: 모든 크롤러 허용, sitemap 위치 명시
- **구조화 데이터**: 선택 사항 — SoftwareApplication JSON-LD로 앱 정보 검색 엔진에 제공 가능 (향후 고려)

### 접근성
- 모든 이미지에 alt 텍스트
- Canvas는 `aria-hidden="true"` (순수 장식)
- 키보드 내비게이션: 다운로드 버튼에 Tab으로 접근 가능
- 색상 대비: WCAG AA 기준 충족 (본문 텍스트 4.5:1 이상)
- prefers-reduced-motion 지원: 패럴랙스/반짝임 비활성화

### 브라우저 지원
- Chrome 90+, Safari 15+, Firefox 90+, Edge 90+
- iOS Safari 15+, Android Chrome 90+
- **인앱 브라우저**: 카카오톡, 네이버, Facebook 등의 인앱 브라우저에서 정상 동작 필수 (주 유입 경로)

## 엣지 케이스

### EC-001: JavaScript 비활성화
- Canvas 별 배경과 패럴랙스 미동작
- HTML/CSS만으로 콘텐츠는 정상 표시 (앱 이름, 설명, 다운로드 버튼)
- 행성 CSS gradient는 JS 없이도 렌더링됨
- `<noscript>` 태그 불필요 — 콘텐츠 자체가 정적이므로

### EC-002: 저사양 디바이스 / 느린 네트워크
- Canvas 별 개수를 디바이스 성능에 따라 조절 (matchMedia로 prefers-reduced-motion 확인)
- 스크린샷 이미지: next/image의 lazy loading + placeholder blur 적용
- 폰트: font-display: swap으로 FOUT 허용 (FOIT보다 나음)

### EC-003: 매우 긴 앱 이름 또는 설명
- 앱 이름: 최대 20자 가정. 넘으면 text-overflow: ellipsis
- 설명: 최대 50자 가정. 넘으면 2줄까지 허용 후 말줄임

### EC-004: 스크린샷 미등록 상태
- 스크린샷이 없는 앱: 행성 그래픽 + 텍스트만 표시
- 스크린샷 영역에 빈 공간 대신 행성 그래픽이 더 강조되는 레이아웃으로 폴백

### EC-005: 앱 스토어 링크 미등록
- 링크가 null/빈 문자열인 경우: 버튼을 "준비 중"으로 표시
- 비활성화 스타일 적용 (opacity 낮춤, cursor: default, 클릭 이벤트 없음)

### EC-006: 인앱 브라우저 제한
- 카카오톡/네이버 인앱 브라우저: `target="_blank"`가 동작하지 않을 수 있음
- 대응: 외부 링크 클릭 시 location.href로도 이동 가능하도록 fallback
- Safari의 경우 window.open 차단 가능 → 직접 링크(<a> 태그)를 우선 사용
- 인앱 브라우저에서 앱스토어 링크 클릭 시:
  - iOS: `https://apps.apple.com/...` → 인앱 브라우저가 App Store 앱으로 전환하는 것이 일반적
  - Android: `https://play.google.com/...` → Play Store 앱으로 전환
  - 별도 딥링크(itms-apps://, market://) 불필요 — HTTPS 링크가 가장 안정적

### EC-011: 해시 앵커 링크 + smooth scroll 충돌
- 페이지 로드 시 URL에 해시가 있으면 (#memory-palace):
  - 브라우저 기본 동작: 해당 id 요소로 즉시 점프 (smooth 아님)
  - JS 하이드레이션 후: 이미 해당 위치에 있으므로 추가 스크롤 불필요
  - 패럴랙스/애니메이션은 현재 스크롤 위치 기준으로 정상 동작
- 히어로 내 바로가기 버튼 클릭 시:
  - `element.scrollIntoView({ behavior: 'smooth' })` 사용
  - `history.pushState`로 URL 해시 업데이트 (페이지 리로드 없이)

### EC-012: 플랫폼 감지 실패 / 하이드레이션 mismatch
- SSR 시 userAgent 없음 → 서버 렌더는 모든 버튼 동등 스타일
- 클라이언트 하이드레이션 후 useEffect에서 감지 → 강조 스타일 적용
- 이때 레이아웃 시프트 방지: 버튼 크기/위치는 변하지 않고, opacity/border 등 시각적 강조만 변경
- 감지 불가 userAgent (봇, 특이한 브라우저) → fallback: 모든 버튼 동등

### EC-013: App Store / Play Store 배지 로딩
- 배지는 SVG로 프로젝트에 포함 (외부 CDN 아님)
- 네트워크 요청 0 → 항상 즉시 렌더링
- SVG를 React 컴포넌트 또는 next/image로 인라인 처리

### EC-007: 뷰포트가 매우 작은 디바이스 (< 320px)
- min-width: 320px 기준으로 디자인
- 320px 미만: 가로 스크롤 발생 가능하나, 사실상 현존 디바이스에서 거의 없음

### EC-008: 뷰포트가 매우 큰 디바이스 (4K 이상)
- max-width: 1280px로 콘텐츠 영역 제한
- 행성/별 배경은 뷰포트 전체 커버
- Canvas 해상도: devicePixelRatio 반영하여 고해상도에서도 선명하게

### EC-009: 앱이 0개인 경우
- 현재 해당 없음 (2개 확정)
- 만약 data/apps.ts가 빈 배열이면: 히어로 + "곧 새로운 앱이 도착합니다" 메시지 + 푸터

### EC-010: prefers-reduced-motion
- 별 반짝임 애니메이션 정지 (opacity 고정)
- 패럴랙스 비활성화 (모든 요소 고정 위치)
- 앱 카드 등장 애니메이션 → 즉시 표시 (transition-duration: 0)

## MVP 제외 사항
- 댓글/게시판/피드백 기능
- 사용자 인증/로그인
- 블로그/포스팅
- 다국어 (한국어 단일)
- CMS / 관리자 페이지
- 검색 기능
- 앱 카테고리/필터
- 다운로드 수 카운터

- RSS 피드
- 다크/라이트 모드 토글 (다크 고정)

## 향후 확장 고려 사항 (구현하지 않되 설계 시 염두)
- 앱 추가 시 `data/apps.ts` 배열에 객체 하나 추가로 완료되어야 함
- 행성 스타일(색상, 크기, 위치)은 앱 데이터에 포함되어 개별 지정 가능해야 함
- 댓글 기능 추가 시 각 앱 섹션 하단에 삽입 가능한 구조
- 애널리틱스 스크립트 삽입 시 layout.tsx 수정만으로 가능해야 함
- 각 앱별 상세 페이지 (`/apps/[id]`) 분리 가능한 구조 — 현재는 앵커 링크로 충분하지만 앱이 10개 이상이면 개별 페이지 필요
- 다국어 지원 시 `data/apps.ts`의 텍스트를 i18n 키로 교체 가능한 구조
- 앱별 OG 이미지 — 특정 앱 앵커로 공유 시 해당 앱에 맞는 OG 이미지 (현재는 전체 사이트 공통 OG 1개)
